    Publishing to B2SHARE (EUDAT)

    This guide explains how DonaDataset is published to B2SHARE using the donadataset CLI. It is aimed at the dataset maintainer.

    1. What is B2SHARE

    B2SHARE is a research data sharing service operated by EUDAT, the European collaborative data infrastructure. It offers secure, long-term storage on servers located within the European Union, and assigns a persistent identifier (an EPIC PID, and a DOI when the hosting community enables it) to every published record. B2SHARE also runs a training/sandbox instance (trng-b2share.eudat.eu) with an identical API, used for testing.

    Records in B2SHARE belong to a community — a curated collection with its own metadata schema and (depending on configuration) its own moderation workflow. You need a community UUID before you can publish anything; request one at b2share.eudat.eu if you don't have one yet.

    2. What B2SHARE allows uploading

    A B2SHARE record can technically hold files of any type, including large binary data — the original intent for this project's B2SHARE mirror was a full "European copy" of the dataset (images, labels, and code). In this project we deliberately use the lighter, linked-record pattern instead (the same one already built for Zenodo): B2SHARE hosts only the metadata and evidence that prove the dataset exists, is verified, and where to find it, while the actual images/labels stay on HuggingFace Hub (see the HuggingFace guide). This avoids B2SHARE's default 10 GB per-record storage quota entirely and keeps publication fast, at the cost of not having a full independent EU-hosted copy of the binary data.

    This is enforced by alternate_identifier in the B2SHARE record's metadata: an explicit link to the HuggingFace dataset page, so anyone landing on the PID/DOI page sees immediately "the data lives here."

    3. What we upload

    donadataset publish b2share prepare does not need a local copy of the HuggingFace export to get the evidence files from — same design as zenodo prepare. It:

    1. Connects to HuggingFace Hub (--repo-id) and downloads the repository live, into its own directory (--output-dir).
    2. Verifies that download against the same checksums/manifest the export was built with (global checksums + internal .tar member hashes).
    3. Picks the small evidence files out of that same downloaded copy (never the .tar shards) and uploads them to a new B2SHARE draft record.

    Unlike Zenodo, B2SHARE does not reserve a PID/DOI at draft-creation time — it's only assigned once the record is actually published (see section 5). So "what we upload" at the prepare stage is: the evidence files already present in the HuggingFace repository (the same ones described in the HuggingFace guide), plus one new report generated by this download-and-verify step, plus a JSON record tying it all together — without an identifier yet.

    4. How we upload it — every file, explained

    Evidence files fetched from the live HuggingFace download

    Same files described in the HuggingFace guide — README.md, LICENSE, CITATION.cff, HuggingFaceHub.yaml, donana.yaml, dataset_info.json, metadata.csv, manifest.csv, manifest-files-sha256.csv, checksums-sha256.txt, validation_report.json, verification_report_local.json — fetched fresh from HuggingFace Hub instead of read from a local pre-upload folder, so what B2SHARE receives matches exactly what's currently public.

    verification_report_downloaded.json

    The result of the live download-and-verify step described above — generated fresh every time b2share prepare runs, and uploaded to B2SHARE as one more piece of evidence, exactly like Zenodo's own version of this file.

    b2share_linked_dataset_record.json

    The result of creating (or publishing) the B2SHARE draft — written locally, not uploaded to B2SHARE itself: record_id, pid/pid_url (empty until published), record_url, the B2SHARE environment (sandbox/production), and a huggingface_verification block summarising the download-and-verify step. Later commands (check-readiness, release, sync-pid) all read this file to know which record they're working with.

    b2share_public_release_readiness_report.json

    Written by check-readiness — not part of the initial prepare upload, but living in the same directory for a complete local audit trail.

    5. Commands to publish

    First-time setup

    1. Log in at b2share.eudat.eu (or trng-b2share.eudat.eu for the sandbox) using your institutional account or ORCID.
    2. Request a community UUID for WildINTEL at b2share.eudat.eu if you don't have one yet — there is no sensible default, every command that touches the network fails with a clear error until this is set. Store it via donadataset publish b2share config set community_id=<uuid>.
    3. Create a personal access token and set it as B2SHARE_TOKEN, or store it once via donadataset publish b2share config set token.

    Publishing a new version

    B2SHARE publication happens after the dataset is already on HuggingFace Hub (see the HuggingFace guide):

    # 1. Create the B2SHARE draft, download+verify HuggingFace Hub live, upload evidence.
#    Uses trng-b2share.eudat.eu (sandbox) by default.
export B2SHARE_TOKEN='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
donadataset publish b2share prepare \
  --repo-id <your-user>/<dataset-slug> \
  --community-id <your-eudat-community-uuid>

# 2. Final read-only safety check before publishing for real
donadataset publish b2share check-readiness --repo-id <your-user>/<dataset-slug>

# 3. Submit the draft for publication
donadataset publish b2share release --repo-id <your-user>/<dataset-slug>

# 4. Detect the PID/DOI (once assigned) and write it into CITATION.cff locally
donadataset publish b2share sync-pid --repo-id <your-user>/<dataset-slug>

# 5. Re-upload to HuggingFace Hub so the updated CITATION.cff is published
donadataset publish huggingface upload

    Or run steps 1, 2, 3, and 4 in one go with pipeline:

    donadataset publish b2share pipeline --repo-id <your-user>/<dataset-slug> \
  --community-id <your-eudat-community-uuid>

    There is no --config flag on any of these commands: they always render the single bundled Jinja2 template (templates/B2SHARE.yaml.j2) using --repo-id/--output-dir/--community-id, whose defaults come from settings.toml — set them once with donadataset publish b2share config set <field>=... (or config wizard) and skip repeating them on every run. The token environment variable name is fixed to B2SHARE_TOKEN in the template (not a setting or a flag) — but its value doesn't have to be exported every session: export B2SHARE_TOKEN=... always wins if set, otherwise it falls back to b2share.token in settings.toml, stored with donadataset publish b2share config set token (hidden input, never echoed or shown by config show).

    ⚠️ community_id has no sensible default — you must request one from EUDAT first. Until you set donadataset publish b2share config set community_id=<uuid>, every command that touches the network will fail with a clear error explaining this.

    ⚠️ Publishing may require moderator approval. Unlike Zenodo (which mints a DOI the moment you publish), B2SHARE communities can be configured to require a moderator to approve new records before a PID/DOI is assigned. b2share release submits the record for publication; b2share sync-pid is the step that later detects whether an identifier has actually been assigned yet, and only then updates your local CITATION.cff — re-run it later if it reports nothing found the first time.
