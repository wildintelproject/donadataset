# Publishing to Zenodo

This guide explains how DonaDataset is published to **Zenodo** using the `donadataset`
CLI. It is aimed at the **dataset maintainer**.

---

## 1. What is Zenodo

[Zenodo](https://zenodo.org) is an open-access research data repository operated by
CERN (Geneva, Switzerland). Researchers use it to archive and share any research
output — datasets, software, papers, presentations — and it assigns a permanent
**DOI** (Digital Object Identifier) to every published record. Zenodo also runs a
**Sandbox** instance (`sandbox.zenodo.org`) with an identical API, used for testing
without touching real DOIs.

Unlike HuggingFace Hub, Zenodo is not designed as a bulk file-hosting service for
machine-learning datasets — it's a **citable, long-term archive**. That's exactly the
role it plays here: it doesn't compete with HuggingFace Hub, it complements it.

## 2. What Zenodo allows uploading

A Zenodo record can technically hold files of any type, including large binary data.
**In this project we deliberately don't use it that way.** DonaDataset follows Zenodo's
own recommended pattern for datasets that are already hosted elsewhere: a
**linked dataset record** — Zenodo hosts only the metadata and evidence that prove the
dataset exists, is verified, and where to find it, while the actual images/labels stay
on HuggingFace Hub (see the [HuggingFace guide](publishing-huggingface.md)).

This is enforced by `related_identifiers` in the Zenodo record's metadata: explicit
links to the HuggingFace dataset page and its file tree, so anyone landing on the DOI
page sees immediately "the data lives here."

## 3. What we upload

Three separate commands split "prepare locally" from "actually talk to Zenodo" —
mirroring the same convention `huggingface prepare`/`upload` already use:

- **`prepare`** never uploads anything to Zenodo. It does **not** need a local copy of
  the HuggingFace export either — instead, it:
  1. Connects to HuggingFace Hub (`--repo-id`) and **downloads the repository live**,
     into its own directory (`--output-dir`).
  2. Verifies that download against the same checksums/manifest the export was built
     with (global checksums + internal `.tar` member hashes).
  3. Creates (or, with `--sync-existing-draft`, re-reads) a Zenodo deposition just to
     **reserve/read a DOI** — no file upload happens yet.
  4. Copies the small evidence files out of that downloaded copy (never the `.tar`
     shards — those stay HuggingFace-only) into `--output-dir`, injecting the reserved
     DOI into that copy's `CITATION.cff` and recomputing that directory's own
     `checksums-sha256.txt` so it stays internally consistent.
- **`upload`** reads exactly what `prepare` staged in `--output-dir` and pushes it,
  unmodified, to the Zenodo deposition's bucket — this is the only command that
  actually talks to Zenodo's file API.
- **`sync-doi`** reflects that DOI back on the HuggingFace Hub side: it copies the
  DOI-patched `CITATION.cff` over the HuggingFace export's own copy, updates the
  "## Zenodo DOI" section in that export's `README.md`, recomputes its
  `checksums-sha256.txt`, and (by default) re-uploads just those three files to
  HuggingFace Hub — see section 4's `--verify-data` box and section 5 for the full
  command sequence.

So "what we upload to Zenodo" is: **the evidence files already present in the
HuggingFace repository** (the same ones described in the
[HuggingFace guide](publishing-huggingface.md#4-how-we-upload-it-every-file-explained)),
with `CITATION.cff` carrying Zenodo's own DOI, plus one new report generated by the
download-and-verify step, plus a JSON record tying it all together. Nothing here is
written by hand — it's all produced by the commands in section 5.

## 4. How we upload it — every file, explained

### Evidence files fetched from the live HuggingFace download

These are the same files already described in the
[HuggingFace guide](publishing-huggingface.md#4-how-we-upload-it-every-file-explained) —
`README.md`, `LICENSE`, `CITATION.cff`, `HuggingFaceHub.yaml`, `donana.yaml`,
`dataset_info.json`, `metadata.csv`, `manifest.csv`, `manifest-files-sha256.csv`,
`checksums-sha256.txt`, `validation_report.json`, `verification_report_local.json` — just
fetched fresh from HuggingFace Hub instead of read from a local pre-upload folder. This
guarantees what Zenodo receives matches exactly what's currently public, not what your
machine happened to have on disk at some earlier point.

`prepare` then copies all of these into `--output-dir` verbatim, **except**
`CITATION.cff` and `README.md`: those two copies get Zenodo's just-reserved DOI
written into them (`CITATION.cff`'s `doi`/`url` fields, or an `identifiers` entry in
Sandbox; `README.md` gets a "## Zenodo DOI" section, same as HuggingFace's own copy
gets via `sync-doi` — see below), and `checksums-sha256.txt` is regenerated for
`--output-dir` right afterwards so it still matches those now-modified files sitting
next to it — meaning Zenodo's own hosted copy of `checksums-sha256.txt` intentionally
differs from HuggingFace's (it describes `--output-dir`'s flat evidence set, not the
full HuggingFace export).

### `<dataset-slug>-camtrap-dp.zip` (optional)

If you've published to [GBIF](publishing-gbif.md) with `gbif prepare` + `gbif
upload`, the resulting Camtrap DP package sits at the root of the
HuggingFace Hub repo alongside the files above. `prepare` picks it up automatically
(by filename pattern, `*-camtrap-dp.zip`, since it's named after GBIF's own
`--dataset-slug`) and includes it as one more evidence file — it's small, structured
metadata with no images in it, so it fits Zenodo's own rule for what belongs here just
as well as `manifest.csv` does. Nothing fails if it isn't there; GBIF publishing is
entirely optional and this is picked up opportunistically.

### `verification_report_downloaded.json`

The result of the live download-and-verify step described above:
`status: "passed"/"failed"`, `repo_id`, the local path the download was written to,
`data_verified` (whether the `data/<split>/*.tar` shards were downloaded and hashed too
— see `--verify-data` below), and the counts/errors from re-checking global checksums
and internal `.tar` hashes. This is generated **fresh every time** `zenodo prepare` runs
(never reused from an older `huggingface download` run), and it is itself uploaded to
Zenodo as one more piece of evidence.

> 💡 **`--verify-data` / `--no-verify-data` (default: off).** Zenodo never uploads the
> `.tar` shards — only the small evidence files — so by default `prepare` doesn't even
> download them from HuggingFace Hub to check them, just the metadata. Pass
> `--verify-data` to additionally download every shard and re-hash its contents against
> `manifest-files-sha256.csv`, for extra assurance that the published images/labels
> themselves (not just the metadata describing them) still match what `huggingface
> prepare` originally wrote — at the cost of a full dataset download every time you run
> this command.

### `zenodo_linked_dataset_record.json`

The **result** of creating the Zenodo deposition — written locally by `prepare`
(`deposition_id`, `record_id`, `reserved_doi`, `doi_url`, `record_url`, the Zenodo
environment, and a `huggingface_verification` block summarising the download-and-verify
step above), then re-written by `upload` once the files are actually pushed and
verified, and also uploaded to Zenodo as a file in the same deposition. Later commands
(`upload`, `check-readiness`, `release`) all read this file to know which existing
deposition they're working with.

### `zenodo_deposition_response.json` / `zenodo_file_verification_report.json` / `zenodo_link_verification_report.json`

Written by `upload` (the step that actually talks to Zenodo's file API) as a
local-only audit trail, not uploaded to Zenodo itself: the raw API response for the
deposition after files were uploaded, a check that every uploaded file's size/hash
matches what Zenodo reports back, and a check that the `related_identifiers` URLs
(the links to HuggingFace) actually resolve.

### `zenodo_publish_response.json` / `zenodo_publication_report.json` / `public_release_readiness_report.json`

Written by the later steps in section 5 (`release`/`check-readiness`) — not part of
`prepare`/`upload`, but living in the same directory for a single, complete audit trail
of the whole release.

## 5. Commands to publish

### First-time setup

1. Create an account at [zenodo.org](https://zenodo.org) (use `sandbox.zenodo.org`
   first to test the whole flow without minting a real DOI).
2. Create a personal access token (**Applications → Personal access tokens**, scopes
   `deposit:write` + `deposit:actions`).
3. Set it as `ZENODO_TOKEN`, or store it once via `donadataset publish zenodo config
   set token`.

### Publishing to Zenodo communities (optional)

`prepare` submits the deposition to one or more [Zenodo
communities](https://zenodo.org/communities) (curated collections — submitting to one
you don't curate yourself leaves the record pending approval; publishing the deposition
itself is unaffected either way). By default that's just `wildintelproject`; change the
full list once (comma-separated) to whatever fits your project:

```bash
donadataset publish zenodo config set communities=camera-traps,biodiversity
```

(pass an empty value, `communities=`, to submit to no community at all)

Every `prepare` run then submits to all of them by default. To narrow a specific run
down to a subset (never to something outside this list — add it to `ZENODO.communities`
first if it's missing), pass `--communities`:

```bash
donadataset publish zenodo prepare --repo-id <your-user>/<dataset-slug> --communities camera-traps
```

This only takes effect when `prepare` creates a **new** deposition — `--sync-existing-draft`
re-reads an already-created draft without resending metadata, so communities set at
creation time aren't changed by a later sync.

### Publishing a new version

Zenodo publication happens **after** the dataset is already on HuggingFace Hub (steps
1–4 of the [HuggingFace guide](publishing-huggingface.md#5-commands-to-publish)), and
the final, irreversible "publish" step happens **after** HuggingFace Hub has been made
public:

```bash
# 1. Create the Zenodo draft, download+verify HuggingFace Hub live, reserve a DOI, and
#    stage the evidence files (DOI already patched into CITATION.cff) in --output-dir.
#    Nothing is uploaded to Zenodo yet. Uses sandbox.zenodo.org by default (see
#    templates/Zenodo.yaml.j2).
export ZENODO_TOKEN='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
donadataset publish zenodo prepare --repo-id <your-user>/<dataset-slug>

# 2. Push exactly what 'prepare' staged to the Zenodo deposition's bucket.
donadataset publish zenodo upload --repo-id <your-user>/<dataset-slug>

# 3. Reflect the DOI on the HuggingFace Hub side: copies the DOI-patched CITATION.cff
#    over the HuggingFace export's own copy, updates the "## Zenodo DOI" section in its
#    README.md, recomputes its checksums-sha256.txt, and re-uploads ONLY those three
#    files to HuggingFace Hub automatically (--no-upload to skip that last part).
donadataset publish zenodo sync-doi --hfh-output-dir <export-dir>

# 4. Make the HuggingFace repository public (if not done already)
donadataset publish huggingface release

# 5. Final read-only safety check before publishing for real. --hfh-output-dir is
#    needed here too: it's how this command finds hfh_publication_report.json,
#    written by 'huggingface release' (step 4) inside that same directory.
donadataset publish zenodo check-readiness --repo-id <your-user>/<dataset-slug> --hfh-output-dir <export-dir>

# 6. Publish the Zenodo draft — IRREVERSIBLE, do this only after step 5 passes
donadataset publish zenodo release --repo-id <your-user>/<dataset-slug>
```

Or run steps 1, 2, 3, 5, and 6 in one go with `pipeline` (step 4, making HuggingFace
Hub public, is the only manual step left — do it whenever suits you, before running
`check-readiness`):

```bash
donadataset publish zenodo pipeline --repo-id <your-user>/<dataset-slug>
```

### The easy way: `wizard`

```bash
donadataset publish zenodo wizard
```

Guides you through the same six steps interactively, one phase at a time. Unlike
`pipeline` (which crashes on the first error and always creates a brand-new draft),
`wizard`:

- Asks for `--repo-id` if you haven't configured one yet (same prompt as
  `donadataset publish huggingface wizard`, shares the same `settings.toml` value), and
  offers to save it so you aren't asked again.
- Detects an existing linked Zenodo draft in `--output-dir` and asks whether to
  synchronise it (`--sync-existing-draft`) instead of always creating a new one.
- Does the HuggingFace Hub re-upload inside step 3 (`sync-doi`) for you automatically,
  instead of leaving it as a manual step you have to remember.
- Warns you before touching `production` if `zenodo.environment` isn't `sandbox`.
- Asks for explicit confirmation before the final publish — the one step that's
  irreversible.
- If any step fails (network hiccup, transient API error...), lets you retry it or abort
  instead of just exiting.

Not to be confused with `donadataset publish zenodo config wizard`, which only edits
`settings.toml` fields and doesn't publish anything.

There is no `--config` flag on any of these commands: they always render the single
bundled Jinja2 template (`templates/Zenodo.yaml.j2`) using `--repo-id`/`--output-dir`
(and `--hfh-output-dir` for `sync-doi`), whose defaults come from `settings.toml` — set
them once with `donadataset publish zenodo config set <field>=...` (or `config
wizard`) and skip repeating `--environment`/etc. on every run. Every command accepts
`--dry-run` (except `download`, which doesn't exist as a publishing step — see
`donadataset publish zenodo download` for *retrieving* an already published record
instead, and `pipeline`, which always runs for real). The token *environment variable
name* is fixed to `ZENODO_TOKEN` in the template (not a setting or a flag) — but its
*value* doesn't have to be exported every session: `export ZENODO_TOKEN=...` always
wins if set, otherwise it falls back to `zenodo.token` in `settings.toml`, stored with
`donadataset publish zenodo config set token` (hidden input, never echoed or shown by
`config show`).

> ⚠️ **`zenodo release` is the only step that actually publishes.** Everything before
> it (`prepare`, `upload`, `sync-doi`, `check-readiness`) only creates/updates a
> **draft** — you can re-run them as many times as you need. Once `release` succeeds,
> the Zenodo record and its files can no longer be edited or removed.
